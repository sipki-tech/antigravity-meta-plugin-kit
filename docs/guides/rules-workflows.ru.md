# Правила и воркфлоу: постоянные ограничители и /slash-команды

*[English](rules-workflows.md) | Русский*

## Rules — постоянные поведенческие ограничители

Правила — это обычный markdown, ограничивающий агента, пока он работает в
своей зоне действия: стиль кода, ограничения API, протоколы безопасности,
тон вывода.

### Где живут правила

| Расположение | Зона действия | Frontmatter |
|---|---|---|
| `GEMINI.md` / `AGENTS.md` в любой директории | эта папка + подпапки (обнаруживаются проходом вверх от CWD до корня репо) | нет — активны всегда |
| `.agents/rules/*.md` в проекте | проект | `trigger: always_on \| model_decision` (наблюдение; `glob` вероятен — уверенность MEDIUM) |
| `plugins/<name>/rules/*.md` | пока плагин включён | не наблюдался |

Иерархия: `GEMINI.md` (специфичен для Antigravity, высший приоритет) →
`AGENTS.md` (портируемый кросс-инструментальный стандарт — тот же файл
работает в Cursor и Claude Code) → файлы конкретных инструментов. Правила
дедуплицируются по резолвнутому пути — правило грузится один раз за
разговор, даже если найдено несколькими путями.

### Как писать работающие правила

- Императив, короткие строки. Каждая строка правил стоит контекстных
  токенов в каждой сессии — файл правил это налог, держите его тощим.
- Группируйте по темам (эталонный плагин везёт `safety.md`, `workflow.md`,
  `output-style.md`).
- Формулируйте проверяемые факты, а не вайбы: «никогда не читай .env» бьёт
  «будь осторожен с секретами».
- Если то же самое энфорсит хук (например danger-guard) — скажите это в
  правиле: правило инструктирует, хук энфорсит.

## Workflows — файлы, становящиеся /slash-командами

Воркфлоу — тонкий `.md` с единственным полем frontmatter:

```markdown
---
description: Планируй нетривиальную работу до кода — нумерованный план со скоупом и рисками
---

1. Загрузи и следуй плейбуку скилла `my-plugin-plan`.
2. Переформулируй задачу: что меняется, зачем, критерии готовности.
3. Запиши план в файл; кода не пиши.
```

- Имя файла → команда: `my-plugin-plan.md` становится `/my-plugin-plan`.
- **`description` обязателен** — это help-текст команды (линт проверяет;
  заметьте: `agy plugin validate` воркфлоу полностью игнорирует — линтер
  этого кита здесь единственный гейт).
- Держите воркфлоу тонкими алиасами на скиллы: плейбук живёт в скилле,
  воркфлоу — поверхность вызова. Не дублируйте логику.
- Расположение: `<project>/.agents/workflows/` (инсталлер зеркалит в
  `.agent/workflows/`, только если `.agent/` уже существует). Скиллы и сами
  автоматически становятся slash-командами, так что воркфлоу оправдан,
  только когда добавляет формулировку или аргументы поверх скилла.

## Подводные камни

- Файл правил, разросшийся в документацию, — агент платит за него каждый
  ход. Справочный материал — в `references/` скилла.
- Воркфлоу без `description` → сломанный help /slash-команды (FAIL линта).
- Дублирование плейбука в скилле и воркфлоу → они разъезжаются.
- Личные предпочтения, закоммиченные в командный репозиторий, — держите их
  в своём глобальном `~/.gemini/GEMINI.md`.

## Чеклист

- [ ] правила: короткие, императивные, сгруппированы по темам, дедуплицированы с хуками
- [ ] AGENTS.md для всего, что должно работать кросс-инструментально
- [ ] у каждого воркфлоу есть description и ссылка на скилл
- [ ] `lint` проходит (rules непусты, frontmatter воркфлоу)
